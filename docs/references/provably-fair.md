# Provably fair algorithm

The transparency of an online casino relies on cryptography. It guarantees that
the result, or the crash point, is predetermined before the betting phase and
does not suffer manipulation based on the amount the players injected into the
round.

## How it works: The hash flow

### 1. Hash chain (Server seed)

Before the casino starts operating, it generates a final secret seed, for
example, randomly. It repeatedly applies the SHA-256 algorithm to this seed a
massive number of times, for example, 2,000,000 times, to create a chain. For
each new game round, the server consumes a hash from this chain by running it in
reverse order.

**Mechanics for the UI:**

1. When the round is in `BETTING_PHASE`, the server openly displays the hash
   generated by the SHA-256 function on the screen, which is the current hash
   of the round. No one can read the crash point through it because reversing an
   SHA-256 hash is impossible.
2. When the round crashes, the server exposes the raw seed that generated that
   hash. This raw seed is from the previous round in the generation sequence, but
   it is the next to be consumed from the database.
3. You can go to a third-party validator, or one provided by the site, input the
   revealed seed, and the algorithm will generate the exact crash point. If you
   apply SHA-256 to the revealed seed, it exactly matches the hash presented
   during the betting phase.

### 2. Client seed

To ensure that the server did not choose a magical hash chain to harm the
players, the system injects a client seed. This can be the hash of the most
recent block on the Bitcoin network for the day, or a public random string from
outside the casino.

## Base calculation

The generated multiplier must follow the house edge formula, for example, 1 to
4%. We recommend the following pseudo-code.

```javascript
import crypto from "crypto";

// The serverSeed is the revealed hash, and the clientSeed is public.
function getCrashPoint(serverSeed, clientSeed) {
  const hash = crypto
    .createHmac("sha256", serverSeed)
    .update(clientSeed)
    .digest("hex");

  // Get the first 52 bits of the hash (first 8 bytes).
  const h = parseInt(hash.slice(0, 13), 16);
  const e = Math.pow(2, 52); // The maximum bit space.

  // If the house wants a 1% house edge, the base calculation is:
  const multiplier = Math.max(1, Math.floor((100 * e - h) / (e - h)) / 100);

  return multiplier; // For example, returns 2.34
}
```

## The API and the game

You must have a `GET /games/rounds/:roundId/verify` route that delivers these
raw parameters after the round ends, so that any auditor can insert them into
the checking script.
